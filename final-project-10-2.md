## Введение

Пишмем небольшую **соц. сеть для лицеистов** (и, возможно, преподавателей) для развития внутреннего сообщества, эдакий ЭлЖур на максималках. Она должна поддерживать:
- Возможность отправки и просмотра сообщений
- Создание групп (для кружков, клубов по интересам и т.п.)
- Создание мероприятий, просмотр общего календаря
- Что угодно ещё, что вы посчитаете полезным и классным

> Проект включает в себя 2 основных части - **фронтенд** и **бэкенд**. За каждую из которых будет выставлена отдельная **констатирующая оценка**.

## Основные требования

### Фронтенд

> Фронтенд должен быть отдельным веб-приложением, разработанным на JavaScript или TypeScript (за TS получите доп. балл). **Можно и скорее даже нужно использовать React или другую библиотеку для построения интерфейсов**

| Требование | Пояснения | Баллы |
| --- | --- | --- |
| Регистрация, аутентификация и страницы регистрации и входа | Для использования приложения пользователь должен зарегистрироваться и войти в систему. Все действия над сущностями при этом должны быть допустимы только для авторизованных пользователей (например, гость не может написать комментарий, а авторизованный пользователь может удалять только собственные комментарии под записями сообществ) | 0-2 |
| Страница отправки и просмотра сообщений | Базовая реализация - как в электронной почте или ЭлЖуре. То есть указывается получатель(-и), тема и текст сообщения. Ещё лучше - сделать чаты в реальном времени, но это дополнительный критерий | 0-2 |
| Лента публикаций и событий | На ней должны отображаться последние новости сообществ, на которые подписан пользователь, и ближайшие события, созданные администрацией | 0-1 |
| Форма создания новости или мероприятия | Должна быть доступна только для администрации. Новость от мероприятия технически отличается только тем, что она не привязана к какой-то дате и времени | 0-1 |
| Страница создания/редактирования группы | На этой странице должна быть форма, позволяющая заполнить информацию о группе и создать её | 0-1 |
| Страница просмотра группы | На ней отображаются все посты, подписчики и прочее | 0-1 |
| Форма создания публикации | Должна быть доступна только для создателя/модератора сообщества | 0-1 |
| Комментарии и оценки публикаций | Возможность оставлять и читать комментарии, оценивать публикации | 0-1 |
| Поиск групп | Пользователь должен иметь возможность найти группу по названию | 0-1 |
| Поиск пользователей | Пользователь должен иметь возможность найти других пользователей по имени, классу и/или по предмету, который он предподаёт | 0-1 |
| Страница просмотра и редактирования профиля пользователя | Собственный профиль можно изменить, чужие - просматривать. Можно, например, добавить возможность просмотра чужого расписания, ссылки на другие соц. сети | 0-1 |
| Валидация форм | Все формы должны проверять значения на корректность и выводить человекочитаемые ошибки при их наличии | 0-1 |
| Обработка серверных ошибок | Если сервер возвращает ошибку, то вы должны в понятном виде отобразить её пользователю, например в виде уведомления | 0-1 |
| Unit- и интеграционные тесты | Тестирование компонентов, функций, страниц и прочего. Балл будет определяться по качеству тестов и проценту покрытия | 0-2 |
| End-to-end тесты | Тесты полных пользовательских сценариев. Этот же критерий есть и для бэкенда, баллы будут учтены в обеих оценках | 0-1 |

### Бэкенд

| Требование | Пояснения | Баллы |
| --- | --- | --- |
| Авторизация | Возможность регистрации, авторизация запросов (т.е., например, удалить запись из группы может только владелец или модератор) | 0-2 |
| Ролевая система доступа | Пользователи должны делиться на учеников, преподавателей и администраторов (либо все преподаватели могут быть администраторами). Администраторы имеют возможность удалять все записи, создавать новости или мероприятия и прочее | 0-1 |
| CRUD-операции для всех сущнсотей | Для всех сущностей должны иметься конечные точки для создания, чтения, обновления и удаления  | 0-2 |
| Хранение данных в БД | СУБД - любая | 0-1 |
| Контейнеризация | Должна быть описана рабочая конфигурация для docker, docker compose или любого другого инструмента | 0-1 |
| Использование инструмента для миграций | Все изменения в структуре вашей БД должны быть описаны в специальном формате для инструмента работы с миграциями, например [Alembic](https://alembic.sqlalchemy.org/en/latest/index.html) | 0-1 |
| Валидация входных данных и обработка ошибок | Все данные, пришедшие на вход вашего сервера, должны валидироваться. В случае некорректности клиенту должна отдаваться ошибка. Также все возможные исключения вашего сервера должны быть обработы | 0-1 |
| Пагинация данных | Все сущности, которых может быть много (пользователи, публикации и т.п.), можно запросить постранично | 0-1 |
| Фильтрация и поиск данных | Сервер должен поддерживать поиск и фильтрацию, если это необходимо для клиентского приложения. Например, если вы делаете поиск постов, то пользователь может ввести название, часть текста или период даты публикации | 0-2 |
| Кэширование | Для потенциально тяжёлых запросов используется Redis или другой инструмент кэширования | 0-1 |
| Unit- и интеграционные тесты | Тестирование компонентов, функций, страниц и прочего. Балл будет определяться по качеству тестов и проценту покрытия | 0-2 |
| End-to-end тесты | Тесты полных пользовательских сценариев. Этот же критерий есть и для бэкенда, баллы будут учтены в обеих оценках | 0-1 |

## Дополнительные требования

> За выполнение дополнительных требований будет ставиться **творческая оценка**. Список ниже - примерный, вы можете реализовать всё, что посчитаете нужным. За любые улучшения, не описанные в основных требованиях, я могу либо добавить баллы к констатирующей, либо выставить творческую. **В общем, любые ваши инициативы будут оценены по достоинству, дерзайте!**

- Смена темы веб-приложения
- Адаптивность под мобильные устройства
- Поддержка [PWA](https://developer.mozilla.org/ru/docs/Web/Progressive_web_apps)
- **Загрузка и отправка файлов** (например, чтобы делиться конспектами с другими учениками)
- Изменение аватарки пользователя
> Для хранения файлов обычно используются специальные хранилища, например Amazon S3 или его аналог - [MinIO](https://www.min.io/)
- **Чаты в реальном времени** (при помощи веб-сокетов)
- Поддержка разметки для текста новостей, мероприятий и публикаций ([WYSIWYG-редактор](https://ru.wikipedia.org/wiki/WYSIWYG), Markdown или подобное)
- Групповые чаты
- Просмотр расписания в профиле ученика
- Страница со статистикой сайта для администраторов ([дэшборд](https://practicum.yandex.ru/blog/chto-takoe-dashbord/) с графиками, например количество активных пользователей за различные периоды)
- Сбор и визуализация метрик
- Ветки комментариев (возможность ответить на комментарий, на комментарий к комментарию и т.д.)
- **Развёртывание на сервере**
- Интернационализация
- CI/CD с линтерами, тестами и автоматическим развёртыванием
- И что угодно другое

## Этапы разработки

### Этап 1. Подготовка к проекту

**Задачи:**
- (Опционально, но крайне желательно) Нарисовать макеты интерфейса веб-приложения
  > Инструменты: [Figma](https://www.figma.com/), [Frame0](https://frame0.app/) или что-то подобное
- Определить пользовательские сценарии. 
  > Можно в графическом формате (просто соединить стрелочками различные страницы/состояния приложения и написать действие, которые при этом совершает пользователь), тогда вам пригодятся макеты из предыдущего пункта. Иначе можно воспользоваться подходом [User Story Mapping](https://practicum.yandex.ru/blog/chto-takoe-user-story-mapping/)
- Спроектировать будущую базу данных (например, в формате ER-диаграммы)
- Определить конечные точки на сервере и контракты данных
  > То есть нужно каким-либо образом (можно текстом, можно графически) представить список всех путей на сервере, куда можно сделать запрос, а так же структуры запроса и ответа сервера
- Выбрать инструменты для реализации: языки программирования, фреймворки, библиотеки, СУБД и так далее

> **Дедлайн**: 07.03.2026. За этот этап ставится **формирующая оценка**

### Этап 2. Начало разработки фронтенда

Задачи:
- Создать и ициниализировать репозиторий
- Установить все необходимые библиотеки
- Настроить необходимые инструменты, как минимум форматировщик и линтер
- Определить структуру проекта. Где вы будете хранить компоненты, где функции, где константы и так далее?
- Разработать следующие интерфейсы:
  - Страница регистрации
  - Страница входа
  - Лента новостей и публикаций с возможностью пагинации
- Написать базовые unit-тесты

> Так как на этом этапе у вас ещё нет сервера, вы должны замокировать его ответы. Можно либо просто описать данные в JSON, либо воспользоваться инструментом по типу [Mirage.js](https://miragejs.com/) или [WireMockи](https://wiremock.org/)

> **Дедлайн**: 21.03.2026. За этот этап ставится **формирующая оценка**

### Этап 3. Начало разработки бэкенда

Задачи:
- Создать и ициниализировать репозиторий
- Установить все необходимые библиотеки
- Настроить необходимые инструменты, как минимум форматировщик и линтер
- Определить структуру проекта
- Описать базовые миграции для БД
- Конечные точки для регистрации и аутентификации
- Написать CRUD для сообществ и публикаций
- Интегрировать бэкенд с фронтендом
- Написать базовые unit-тесты

> **Дедлайн**: 04.04.2026. За этот этап ставится **формирующая оценка**

### Этап 4. Доработка проекта

На этом этапе вы дорабатываете проект до его финального состояния. **По завершению ставится 2 констатирующих оценки: одна за фронт, другая - за бэк**

> **Дедлайн**: 16.05.2026
