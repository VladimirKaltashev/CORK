# CORK Agent Rules

Правила для Kimi / OpenCode / другой coding-модели / человека-разработчика, который работает над CORK Social.

Главный принцип: **не просто писать код, а сохранять продуктовую идею, архитектуру и стиль проекта**.

---

## 0. Что такое CORK Social

CORK Social — не «соцсеть достижений школьников».

CORK Social — это **суд интернета над всем, что претендует на крутость**.

Пользователь может принести своё достижение, чужую находку, фейл, понт, проект, организацию или абсурдную историю. Сообщество решает: **👑 корона или 🤡 клоун**.

Ключевая формула:

```text
Claim → Proof → Verdict → Reputation
```

Ключевая рамка:

```text
Мы судим заявку, а не уничтожаем человека.
```

CORK должен быть смелым, дерзким, мемным, но не превращаться в травлю и мусорку.

---

## 1. Перед началом каждой сессии

Перед любыми изменениями разработчик обязан прочитать актуальные документы проекта.

Минимум:

```text
README.md
product.md
CORK_AGENT_RULES.md
```

Также проверить, есть ли в репозитории:

```text
reports/
docs/
supabase/
src/styles/theme-tokens.css
src/styles/theme-components.css
src/styles/themes/acid.css
```

Если есть свежие отчёты в `reports/`, прочитать последние 1–3 отчёта перед началом работы.

Перед кодом нужно понять:

1. Что уже реализовано.
2. Что было изменено в последней сессии.
3. Какие ограничения уже были согласованы.
4. Не противоречит ли новая задача продуктовой идеологии.
5. Не дублирует ли новая задача уже существующую архитектуру.

---

## 2. Перед началом работы написать план

Перед изменением файлов нужно коротко описать план:

```text
Цель:
Что будет изменено:
Какие файлы будут затронуты:
Что НЕ будет тронуто:
Риски:
Проверки:
```

Нельзя начинать большую миграцию без явного согласия пользователя.

Нельзя расширять задачу «заодно».

Если задача маленькая — фиксить только её.

## 3. Продуктовые правила

### 3.1. Не сужать CORK обратно до «достижений»

Старый фокус «достижения школьников» слишком узкий.

Новая модель шире:

```text
Claim / Заявка / История
```

Типы claim:

```text
achievement — реальное достижение
flex — понт / заявка на крутость
fail — фейл
absurd — абсурд / мем
scout_find / discovery — найденное крутое или полезное
organization — достижение команды, школы, проекта, компании
debate — спорная заявка
```

Achievement — только один из типов claim.

### 3.2. Различать author и subject

Важно:

```text
author = кто опубликовал
subject = о ком / о чём claim
```

Пользователь может публиковать не только про себя.

Например:

```text
author: Вася
subject: команда школьников / проект / официант / организация
claim: «они сделали X»
```

### 3.3. Сохранять роли экономики

В CORK есть минимум 3 роли:

```text
Performer — тот, кто сделал
Scout — тот, кто нашёл и принёс
Judge — тот, кто вынес вердикт
```

Разработчик не должен проектировать новые фичи так, будто важен только автор своих достижений.

### 3.4. Скаутская экономика обязательна

Пользователь должен получать ценность за то, что он нашёл и принёс интересный claim.

Будущие механики должны учитывать:

```text
Scout Score
Top scouts
Best finds
King Hunter
Clown Hunter
Thread Starter
```

### 3.5. Комментарии — часть суда

Комментарии не должны быть просто «чатом под постом».

Идеальная модель:

```text
Аргументы за 👑
Аргументы за 🤡
Пруф?
Общие реплаи
Лучший аргумент
```

## 4. Идеология и тон

CORK не должен звучать как:

```text
мотивационная соцсеть
портфолио достижений
школьный проект
нейрослопный SaaS
стерильная платформа прогресса
```

CORK должен звучать как:

```text
дерзко
мемно
остро
соревновательно
немного опасно
но не тупо токсично
```

Разрешённые формулировки:

```text
Не лайк. Не дизлайк. Вердикт.
Кидай заявку на величие.
Толпа решит: корона или клоун.
Судим заявку, не человека.
Принеси claim. Докажи. Получи вердикт.
```

Нежелательные формулировки:

```text
развивающее сообщество
платформа для мотивации
делитесь своими успехами
дружелюбная среда достижений
расти вместе с друзьями
```

---

## 5. Безопасность и модерация

Публикация должна быть быстрой. Не делать обязательную премодерацию для всех claims, если пользователь явно не попросил.

Правильная модель:

```text
пост публикуется сразу
↓
имеет статус unverified / not reviewed
↓
модерация постфактум
↓
жалобы / ручная проверка / будущая AI-модерация
```

Статусы:

```text
unverified — опубликовано, но не проверено
verified — проверено / пруф норм
disputed — спорно / жалобы
hidden — скрыто модерацией
rejected — удалено модерацией
```

Критическое правило:

```text
🤡 относится к claim, а не ко всей личности человека.
```

Не строить интерфейс так, будто CORK поощряет травлю конкретного человека.

## 6. Архитектурные правила фронтенда

### 6.1. Не плодить theme-specific компоненты

Запрещено без отдельного согласования:

```text
FeedPageAcid.tsx
HeaderAcid.tsx
ReactionBarAcid.tsx
ProfilePageAcid.tsx
FriendsPageAcid.tsx
```

Правильно:

```text
один FeedPage
один Header
один ReactionBar / VerdictBar
один ProfilePage
один FriendsPage
```

Темизация делается через CSS variables и `cork-*` классы.

### 6.2. Темы через токены и CSS

Основные файлы:

```text
src/styles/theme-tokens.css
src/styles/theme-components.css
src/styles/themes/acid.css
```

Позже новые темы добавляются как:

```text
src/styles/themes/royal.css
src/styles/themes/blueprint.css
src/styles/themes/obsidian.css
src/styles/themes/candy.css
```

Но не через копирование React-компонентов.

### 6.3. Использовать cork-* классы

Предпочтительно:

```text
cork-page
cork-shell
cork-card
cork-panel
cork-btn
cork-btn-primary
cork-tab
cork-tag
cork-avatar
cork-verdict-bar
cork-verdict-track
cork-verdict-king
cork-verdict-clown
```

Не разносить стили по сотням Tailwind color classes, если элемент уже должен быть частью дизайн-системы.

### 6.4. Не злоупотреблять !important

`!important` допускается только как временный точечный override для темы, если иначе ломается specificity.

Нельзя строить тему на огромном списке Tailwind overrides.

Если появляется много `!important`, нужно остановиться и предложить нормализацию `theme-components.css`.

## 7. Правила по React / TypeScript

1. Не создавать дублирующие компоненты без необходимости.
2. Не делать большие рефакторы без запроса.
3. Не менять схему данных без миграции и описания причины.
4. Не ломать существующие tests.
5. Не оставлять `console.log`, debug UI, временные заглушки без пометки.
6. Не использовать `any`, если можно типизировать нормально.
7. Не делать silent return в важных actions без ошибки для пользователя.
8. При изменении store/actions проверять все места вызова.

Особенно важно:

```text
friends store
auth store
reaction budget
achievement/claim creation
moderation status
```

---

## 8. Правила по reactions / verdict

Вердикт — ядро продукта.

Нельзя ухудшать или удалять:

```text
👑 / 🤡 реакции
бюджет голосов
verdict bar
ratio / score
```

Базовая текущая экономика:

```text
10 голосов в неделю
👑 стоит 1
🤡 стоит 2
остаток сгорает
```

Будущая возможная механика:

```text
можно вложить несколько голосов в один claim
stake / сила вердикта
ранний точный судья получает Judge Score
```

Но это требует отдельного проектирования БД и UI. Не внедрять «между делом».

Verdict bar должен быть:

```text
полноширинный
видимый
понятный
без дублирования счётчиков, если они уже есть в кнопках
корректный при 0/0, 1/0, 0/1, 1/1, 3/1
```

## 9. Правила по челленджам

Челленджи в CORK — не скучные задания.

Челлендж = временная арена / ветка с дедлайном, участниками, вердиктом и рейтингом.

Типы будущих челленджей:

```text
Skill Arena — сделай сам
Scout Hunt — найди самое крутое
Fail Hunt — найди самый легендарный фейл
Useful Hunt — найди самое полезное
Organization Clash — команды/школы/организации против друг друга
Hot Seat — спорная заявка: достижение или кринж?
```

Не делать челленджи как «просто список задач».

## 10. Правила по документации

После каждой содержательной сессии нужно обновить документацию.

Обязательно:

```text
README.md — если изменились запуск, архитектура, env, основные фичи
product.md — если изменились продуктовая концепция, механики, роли, roadmap
CORK_AGENT_RULES.md — если появились новые правила работы
```

Если изменение только CSS-bugfix, `product.md` можно не трогать.

Если изменение меняет смысл продукта, user flow, БД или roadmap — `product.md` обновить обязательно.

## 11. Отчёты

После каждой сессии нужно создать отчёт в папке:

```text
reports/
```

Формат имени:

```text
reports/session-YYYY-MM-DD-short-topic.md
```

Пример:

```text
reports/session-2026-06-06-verdict-bar-fix.md
reports/session-2026-06-07-product-model-v2.md
```

Отчёт должен содержать:

```text
# Отчёт — [тема]

## Дата

## Цель

## Что изменено

## Какие файлы затронуты

## Что НЕ трогали

## Проверки
- npm run lint
- npx tsc -b --noEmit
- npm test
- npm run build

## Коммит

## Риски / оставшиеся проблемы

## Что проверить в браузере

## Следующий рекомендуемый шаг
```

Если задача была только исследовательской и код не менялся, всё равно создать отчёт, но указать:

```text
Код не менялся.
Коммит не создавался.
```

## 12. Git rules

После завершения задачи разработчик должен:

```bash
git status
git diff
npm run lint
npx tsc -b --noEmit
npm test
npm run build
git add .
git commit -m "type: short description"
git push
```

Коммитить самостоятельно, если пользователь не сказал обратное.

Не пушить, если:

```text
lint/typecheck/build падают
тесты падают без объяснения
есть незавершённый конфликт
есть случайные debug-файлы
есть секреты / ключи / .env
```

Сообщения коммитов:

```text
feat: add scout claim model proposal
fix: restore verdict bar layout
docs: rewrite product manifesto
refactor: migrate profile cards to cork classes
style: improve acid button contrast
```

## 13. Проверки перед финальным отчётом

Минимальный набор:

```bash
npm run lint
npx tsc -b --noEmit
npm test
npm run build
```

Если какой-то шаг не может быть выполнен, нужно явно написать почему.

Нельзя писать «всё готово», если проверки не запускались.

---

## 14. Что нельзя делать без отдельного согласования

Нельзя без явного разрешения:

```text
менять схему БД
добавлять новые темы
делать массовую миграцию всего проекта
удалять существующие фичи
переименовывать основные routes
менять экономику голосов
добавлять AI-модерацию
добавлять оплату / premium
переименовывать продукт
заменять дизайн-систему
```

Если задача требует этого — сначала написать предложение и дождаться подтверждения.

---

## 15. Как принимать решения, если задача неясна

Если задача неясна, не угадывать и не делать большую реализацию.

Нужно ответить в одном из форматов:

```text
Я вижу 2 варианта реализации...
Рекомендую вариант A, потому что...
Для кода нужно подтвердить...
```

Если можно безопасно сделать маленький шаг — сделать маленький шаг.

Если шаг влияет на продуктовую модель — сначала proposal.

## 16. Текущие приоритеты продукта

Актуальный вектор:

```text
1. Укрепить CORK Social как claim/verdict/scout платформу.
2. Не возвращаться к узкой модели «достижения школьников».
3. Добавить comments как часть суда.
4. Продумать и внедрить Scout Economy.
5. Переделать создание поста из achievement-only в claim creation.
6. Челленджи проектировать как временные арены.
7. Темы и визуальные улучшения делать после стабилизации ядра.
```
